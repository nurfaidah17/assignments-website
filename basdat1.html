Tugas 1 Basis Data II


Apa itu Query?

Query adalah sebuah ekspresi bahasa yang menggambarkan data yang akan didapatkan kembali dari sebuah database. Query adalah perintah-perintah untuk mengakses data pada sistem basis data. Dalam hubungannya dengan optimisasi query, seringkali diasumsikan bahwa query-query tersebut dinyatakan dalam sebuah dasar-dasar isi dan sekumpulan cara orientasi, yang memberikan optimizer pilihan-pilihan diantara alternatif prosedur-prosedur evaluasi. Query merupakan bagian dari basis data. Untuk itu sistem basis data memerlukan metode terbaik untuk menjawab query. Pencarian strategi dalam pemrosesan query sangat penting agar pemrosesan dapat dilakukan secara cepat. Pada beberapa Data Base Management System (DBMS), pemroses query memilih dari sekumpulan strategi dengan berbasiskan heuristik tertentu.

Optimasi merupakan suatu langkah untuk mengoptimalkan waktu menjadi lebih efisien. Ketika sebuah query diberikan pada sistem basis data, optimasi penting dilakukan untuk memilih strategi yang efisien untuk mengevaluasi ekspresi relasi yang ditentukan.

Optimasi Query adalah suatu proses menganalisa query untuk menentukan sumber-sumber apa saja yang digunakan oleh query tersebut dan apakah penggunaan dari sumber tersebut dapat dikurangi tanpa merubah output. Atau bisa juga dikatakan bahwa optimasi query adalah sebuah prosedur untuk meningkatkan strategi evaluasi dari suatu query untuk membuat evaluasi tersebut menjadi lebih efektif. Optimasi query mencakup beberapa teknik seperti transformasi query ke dalam bentuk logika yang sama, memilih jalan akses yang optimal dan mengoptimumkan penyimpanan data. Tujuan dari optimasi query adalah menemukan jalan akses yang termurah untuk meminimumkan total waktu pada saat proses sebuah query. Untuk mencapai tujuan tersebut, maka diperlukan optimizer untuk melakukan analisa query dan untuk melakukan pencarian jalan akses.


Diagram Alir Query


  1. Scanner, melakukan identifikasi (pengenalan) token-token seperti SQL keywords, attribute, dan relation name.
  2. Query Parser, melakukan verifikasi terhadap validitas dari statemen SQL, kemudian menterjemahkan Query kedalam struktur internal menggunakan kalkulus relasional.
  3. Query Oprimizer, mencari ekspresi terbaik dari beberapa ekspresi aljabar yang berbeda. Kriteria yang digunakan adalah "Murahnya Biaya"
  4. Code Generator/Interpreter, menerjemahkan Query Plan yang dihasilkan oleh Query Optimizer menjadi Query Code untuk kemudian dikirimkan ke Query Processor
  5. Query Processor, mengeksekusi Query Code yang didapat dari Code Generator.

Cost-Based Query Optimization

Biaya (Cost) adalah waktu yang dibutuhkan untuk mendapatkan semua jawaban. Hampir semua DBMS menggunakan cost-based optimizer untuk mengoptimalkan query.

Tahapan-tahapan pada Cost-Based Query Optimization:

  1. Parsing
  2. Transformasi
  3. Implementasi
  4. Perencanan pemilihan skenario query berdasarkan estimasi biaya

Estimasi Biaya Query

Strategi yang dipilih dalam evaluasi query tergantung pada estimasi biaya dari masing-masing strategi yang ada. Optimizer Query akan membuat informasi statistik yang disimpan dalam katalog DMBS untuk memperkirakan biaya dari sebuah rencana query.

  • nr = Jumlah record dalam tabel r
  • br = Jumlah block yang menampung data pada tabel r
  • sr = Ukuran setiap record dari tabel r (dlm Byte)
  • fr = blocking factor/jumlah record yg dapat ditampung oleh 1 blok
  • V(A,r) = Jumlah nilai unik yang ada pada atribut A di tabel r
  • SC(A,r) =rata-rata jumlah record yang memenuhi kondisi kesamaan terhadap atribut A

Biaya evaluasi query dapat diukur dari banyaknya sumber daya sistem yang digunakan, misl: pengaksesan disk, waktu CPU dan waktu komunikasi


Evaluasi Ekspresi

Mengevaluasi query dilakukan bagian-perbagian secara berurutan.

  • Materialisasi

    • Materialization mengevaluasi 1 operasi pada satu waktu, dimulai pada level yang paling rendah kemudian dilanjutkan ke atasnya. Cost untuk menulis result pada disk dan membaca kembali cenderung besar. Materialisasi ini menampung hasil operasi menggunakan tabel temporer (dimaterialkan) yang kemudian digunakan untuk evaluasi level berikutnya.

  • Pipeline

    • Pipelining mengevaluasi banyak operasi secara bersamaan, melewati hasil satu opeasi ke berikutnya. Pipelining jauh lebih murah dari pada materialisasi, karena tidak perlu untuk menyimpan sementara hubungan ke disk. Pipelining mungkin tidak selalu bisa digunakan pada sort, hash-join. Agar pipelining menjadi efektif, menggunakan algoritma evaluasi yang menghasilkan output tupel bahkan sebagai tupel diterima untuk masukan kepada operasi. Pipeline mengganti tabel temporer dengan suatu pipeline operasi (berupa buffer). Penggunaan pipeline akan menghilangkan biaya pembacaan dan penulisan tabel-tabel temporer. Pipelines dapat diekseskusi dalam dua cara yaitu demand driven dan producer driven.
